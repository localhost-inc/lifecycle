## Product Vision

1. `Lifecycle` is a desktop-first workspace runtime that turns a repository plus `lifecycle.json` into a reproducible, disposable environment. The primary interface is a Tauri desktop app (Rust backend + React webview). A thin Cloudflare Worker handles webhook ingress and preview URL gateway. Mobile app (React Native) is the future monitoring surface. Workspaces execute via pluggable `WorkspaceProvider` implementations — `CloudWorkspaceProvider` (cloud sandboxes with preview URLs) and `LocalWorkspaceProvider` (local machine with Tauri process supervision). Both ship in V1. Local workspaces are private to the machine — only cloud workspaces are visible to the organization. Local workspaces can be forked to cloud for sharing.
2. The core V1 product loop is: `open app -> add project -> start workspace -> run agent -> (optional: sign in -> fork to cloud -> share preview -> create PR)`.
3. Primary value is speed to a healthy, shareable runtime: fastest path from "I need this running" to "here is a live environment and PR".
4. Lifecycle is agent-agnostic infrastructure ("where agents run"), not a competing coding-agent product.
5. Shared workspace state, deterministic setup/reset, and preview URLs are core; Slack/Linear issue loops are expansion after the core desktop path is solid.

## Platform Model

### Control Plane

The control plane is split into two layers: a local layer that requires no network or auth, and a cloud layer for collaboration.

#### Local Control Plane (Tauri)

1. **Tauri Rust backend** is the local state store and process supervisor:
   - local project, workspace, and terminal state stored in Tauri-managed SQLite (`rusqlite`)
   - no auth, no network required — app works immediately on first launch
   - workspace status field is the single source of truth for in-flight local operations
   - transitional states (`creating`, `starting`, `resetting`, `sleeping`, `destroying`) act as implicit mutex

#### Cloud Control Plane (Convex)

1. **Convex** is the cloud control plane (active only when user is signed in):
   - cloud workspace state, org state, activity, collaboration
   - auth verification via WorkOS JWT (`ctx.auth.getUserIdentity()`)
   - Convex HTTP actions for GitHub webhook ingress
   - RBAC enforcement in Convex functions
   - real-time subscriptions for cloud workspace state
2. **Thin Cloudflare Worker** (not a web app):
   - GitHub webhook signature verification + forwarding to Convex HTTP actions
   - Preview URL gateway (wildcard Worker + `proxyToSandbox()`)
   - No UI, no auth middleware, no server routes

### Execution Plane

The execution plane is the adapter layer between the control plane and the environment where workspaces actually run. It is modeled as a `WorkspaceProvider` interface — the primary extensibility seam for workspace lifecycle operations. Workspace mode is selected **per-workspace** at creation time and stored as `workspace.mode`.

Full interface and provider details: [`reference/workspace-provider.md`](reference/workspace-provider.md)

## Project and Workspace Model

1. `project`:
   - universal root entity — every workspace belongs to a project
   - represents a directory on disk with a `lifecycle.json` runtime config
   - optionally linked to a `repository` for VCS integration (GitHub push, fork-to-cloud, PR creation)
2. `repository`:
   - thin VCS identity table — provider, owner, name, installation credentials
   - a project links to a repository via `repository_id` when the user connects to GitHub
3. `workspace`:
   - isolated execution instance for a `project` at a specific `source_ref`
   - backed by a dedicated Git worktree checked out at target ref/SHA
   - contains run state, logs, and test history
   - per-service runtime state tracked in `workspace_service` table
4. Naming in V1:
   - canonical noun is `workspace` across CLI, Convex functions, storage schema, and desktop app surfaces
   - avoid legacy alternate naming in new contracts and docs

## Blueprint of the World

1. `project`:
   - universal root entity — a directory on disk with optional `lifecycle.json`
   - exists locally first (Tauri SQLite), optionally synced to Convex when user signs in
   - optionally linked to a `repository` for VCS integration
2. `repository`:
   - thin VCS identity record (provider, owner, name, installation credentials)
   - a project links to at most one repository
3. `organization`:
   - tenancy boundary for identities, projects, and policies
   - projects belong to an organization when the user signs in
4. `workspace`:
   - isolated execution instance of a project at a specific ref/SHA
   - always scoped to a project, regardless of mode
5. `workspace_service`:
   - per-service runtime state for a workspace (exposure, port overrides, preview state)
6. `terminal`:
   - PTY-backed terminal session scoped to one workspace; optionally runs a coding agent harness
7. Relationship chain:
   - `organization -> project[*]` (when signed in)
   - `project -> repository?` (optional VCS link)
   - `project -> workspace[*]`
   - `workspace -> terminal[*]`
8. Expansion-scope entities (see `docs/plans/lifecycle/expansion/`):
   - `thread` + `message`: conversation continuity for Slack/Linear/CLI prompts
   - `organization_image`: custom sandbox base image registry

## Workspace Lifecycle

1. `create`: create Git worktree for ref, run project `setup` steps, initialize `workspace_service` records from project services config, start all defined services, run smoke boot check.
2. `run`: restart all workspace services using project-level `services` config plus workspace-level service overrides from `workspace_service` records.
3. `reset`: restore baseline fixtures and restart all defined services.
4. `sleep`: automatic via `idle_timeout_minutes`; persists workspace state (mechanism varies by provider), stops services, records `sleeping` state. Wake is automatic on access.
5. `destroy`: terminate workspace, revoke credentials, and clean metadata.
6. Expansion-scope: `test` (run selected suite and stream output), `snapshot` (capture known-good state for quick restore).

## Terminal Concept

A terminal session is a PTY process running inside a workspace execution environment. It may optionally run a coding agent harness (Claude Code, Codex, etc.) or be a plain shell. The desktop app terminal panel is a terminal client connected to the workspace PTY; the process runs inside the execution environment.

Full terminal model, agent CLI integration, and PTY architecture: [`milestones/m3.md`](milestones/m3.md)

## Cross-Mode Collaboration

Local workspaces are private to the machine. They are NOT visible to teammates in the organization workspace list. Only cloud workspaces appear in the org workspace list. Fork-to-cloud is the only way to make local work visible to the organization.

#### Cloud Workspace Visibility

Cloud workspaces appear in the organization workspace list. Teammates see real-time status, services, and terminal session state for cloud workspaces via Convex reactive queries. Mode badge (`cloud`) identifies the execution context.

#### Fork-to-Cloud (the bridge)

Fork-to-cloud is the bridge between local and cloud. A user working locally can fork their workspace to cloud to share it with teammates. The forked workspace runs full `create → setup → start` lifecycle — it's a fresh cloud workspace at the same ref, not a migration.

- Fork flow: stash-commits dirty working tree to a temporary branch (`lifecycle/fork/<short-id>`), pushes it, and creates a cloud workspace at that branch via Convex mutation. The `includeUncommitted` option is especially important here — local work may have significant uncommitted state.
- Desktop app: "Fork to cloud" action on any local workspace (requires sign-in and GitHub App). Checkboxes: "Destroy local source after fork" and "Include uncommitted changes" (hidden when working tree is clean).
- Activity event (cloud): `workspace_forked` with `{ source_mode: "local", target_workspace_id, target_mode: "cloud", source_destroyed, included_uncommitted }`
- Common flow: you've been iterating locally → sign in → fork to cloud so teammate gets a shareable preview URL. Optionally include uncommitted changes and/or destroy local source.

#### Cloud-to-Local Fork

Any org member can also fork a cloud workspace locally for faster iteration with instant terminal, no provisioning latency. This creates a local workspace at the same ref.

- Desktop app: "Fork locally" action on any cloud workspace.
- The local workspace is private — it does not appear in the org workspace list.

## Live Preview

Four guarantees that define preview behavior:

1. Preview URL remains stable for the life of the workspace (including hot reload and wake cycles).
2. Users opening a sleeping workspace preview get a deterministic "waking workspace" response rather than a generic 404.
3. Route updates are near-real-time so teammates can watch endpoint changes while development is in progress.
4. HTTP/HTTPS and WebSocket (`ws/wss`) upgrade are supported; long polling and SSE pass through without gateway buffering breakage.

Full preview lifecycle, Cloudflare routing model, and auth details: [`milestones/m5.md`](milestones/m5.md)

## Core User Journeys (V1)

### Journey 0: First-Time Desktop App Onboarding (Core Product Entry)

1. Trigger:
   - user downloads and opens the Lifecycle desktop app
2. Expected flow:
   - app opens instantly, no login screen
   - user clicks "Add project" and selects a directory from the filesystem
   - app reads and validates `lifecycle.json` from the directory
   - sidebar shows project name with config status (valid/invalid/missing)
   - user clicks "Start" to create their first local workspace
3. Failure handling:
   - invalid `lifecycle.json` returns field-level validation with remediation guidance
   - missing `lifecycle.json` shows "No config found" with init prompt
4. UX outcome:
   - user reaches workspace-ready state in under 5 minutes with no account, no network, and no setup friction

### Journey 1: Create First Workspace and Start First Agent Session

1. Trigger:
   - user clicks `Create workspace` in desktop app for their project
2. Expected flow:
   - Tauri backend reads project's `lifecycle.json` and validates runtime config
   - creates isolated Git worktree for requested ref
   - provisions workspace via selected provider (local machine or cloud sandbox)
   - executes project `setup` once
   - resolves secret references and injects required values at runtime
   - starts all defined services (for example `api + postgres + redis`)
   - waits for health checks, marks workspace `ready`, and opens workspace detail
   - user starts first terminal session and attaches in desktop app
3. Failure handling:
   - if validation fails, request is rejected synchronously with field-level errors
   - if setup/service start fails, workspace moves to `failed` with last failed step + logs
   - if terminal start fails, user sees typed failure with retry action
4. UX outcome:
   - user has first active terminal session in a `ready` workspace from desktop app only

### Journey 2: Create First PR from Workspace

1. Trigger:
   - user requests `Create PR` from workspace branch in desktop app
2. Expected flow:
   - control plane validates head/base refs and commit diff
   - PR is created via GitHub App permissions
   - desktop app returns PR URL, number, and status context
3. Failure handling:
   - no-diff and permission failures return typed errors with suggested next action
4. UX outcome:
   - user completes first code-delivery loop (`terminal -> commit -> PR`) without leaving desktop app

### Journey 3: Daily Developer Loop in Existing Workspace

1. Trigger:
   - developer opens desktop app workspace console and selects a `ready` workspace
2. Expected flow:
   - inspect service status and logs
   - run `reset` when state drifts
   - restart services via `workspace run`
   - share preview URL with teammates via `workspace service set --share on`
3. Failure handling:
   - conflicting actions are blocked with clear state-transition errors
   - stale or unhealthy runtime prompts `run` restart suggestion
4. UX outcome:
   - fast `diagnose -> reset/run -> share` loop without leaving the desktop app workspace console

### Journey 4: Share and Reproduce a Failure

1. Trigger:
   - developer needs a teammate to look at their workspace state
2. Expected flow (cloud workspace):
   - developer runs `lifecycle workspace service set --service api --share on`
   - lifecycle updates service exposure in workspace and returns preview URL
   - teammate opens shared preview URL and sees live workspace state
3. Expected flow (local workspace):
   - local workspaces are NOT visible to teammates — developer must fork to cloud first
   - developer clicks "Fork to cloud" (optionally including uncommitted changes)
   - cloud workspace reaches `ready` with shareable preview URL
   - teammate opens preview URL or forks the cloud workspace locally for faster iteration
4. Failure handling:
   - expired workspace returns explicit TTL error with recreate action
5. UX outcome:
   - fork-to-cloud is the bridge: local iteration → one-click share → teammate has live preview
   - teammate can fork cloud workspace locally for faster debugging

### Journey 5: Cross-Mode Handoff

1. Trigger:
   - developer has been iterating locally and wants to share progress or hand off to teammate
2. Expected flow:
   - developer clicks "Fork to cloud" in desktop app workspace operations panel
   - if uncommitted changes exist, developer optionally checks "Include uncommitted changes" — this stash-commits to a temporary branch and pushes
   - control plane creates cloud workspace at that ref, runs setup + services
   - cloud workspace reaches `ready` with shareable preview URL
   - teammate sees new cloud workspace in their desktop app, opens preview, attaches terminal session
   - developer optionally destroys local workspace (via `destroySource` option on fork)
3. Failure handling:
   - if cloud workspace create fails, source workspace is unaffected (source is only destroyed after target reaches `ready`)
   - fork failure surfaces typed error with suggested next action
4. UX outcome:
   - fast local iteration → one-click cloud handoff → teammate has live preview + terminal access

### Expansion-scope journeys (see `docs/plans/lifecycle/expansion/`)

- **Issue-driven workspace from Slack/Linear** (threads.md)
- **Business/user prompt loop** (threads.md)

### Journey-Derived Product Requirements (Wedge)

1. Project runtime contracts (`lifecycle.json` `setup`, `services`) are required and schema-validated.
2. Workspace creation must be deterministic and explain every failed step.
3. Desktop app workspace console must optimize for the repeat loop: inspect logs, reset/restart, share, and PR handoff.
4. Share flows must include enough context to reproduce failures without extra setup.
5. Per-workspace service overrides must support share toggles and port overrides without mutating project config.
6. First-time onboarding must be complete in the desktop app (add project -> workspace create -> terminal start) without manual API calls or account creation. Sign-in and cloud features are opt-in upgrades.
7. Every failure surfaces a typed error with what happened, why, and suggested next action.

2. Left panel (`NAV`):

   - nested terminal session rows under each workspace

3. Center panel (`SESSION`):
   - workspace header (`repository / ref`, status badge, terminal tabs)

### Interaction Model

1. Core desktop app actions:
   - switch active organization
   - create/select workspace
   - run/restart runtime
   - reset workspace
   - destroy workspace
   - toggle service sharing and copy preview URL
   - create/attach terminal session
   - fork workspace to different mode (with optional source destroy)
2. Safety interactions:
   - `destroy` requires explicit confirmation dialog with workspace identifier
   - `reset` requires explicit confirmation before execution
3. Session behavior:
   - switching sessions detaches/attaches stream without killing process
   - detached sessions retain output history for reattach

### Auth in Desktop App

> Full auth spec in [`milestones/m4.md`](milestones/m4.md).

1. Auth provider: WorkOS AuthKit via Device Authorization Flow.
2. Sign-in: device flow opens system browser for WorkOS-hosted AuthKit page (GitHub OAuth primary, SSO for enterprise orgs). User confirms in browser; app receives tokens.
3. Token storage: OS keychain via Tauri secure storage — no cookies, no session table.
4. Organization context: Convex `ctx.auth.getUserIdentity()` returns `organizationId`, `role`, `permissions`.
5. Organization switcher: updates active organization context; scopes all Convex queries.
6. Auth is optional: local project and workspace features work without sign-in. Cloud features (org workspace list, fork-to-cloud, preview sharing, PR creation) require sign-in.
7. Auth is unified: desktop app and CLI both use device flow → WorkOS JWT → Convex verification.

### Data and Transport Bindings

**Local mode (no auth required):**
1. Local project, workspace, and terminal state is stored in Tauri SQLite and accessed via Tauri IPC commands from the React webview.
2. Workspace status updates are pushed from Rust backend to webview via Tauri events (no polling).
3. Terminal: xterm.js in Tauri webview, connected via Tauri IPC to Rust-managed PTY. No WebSocket needed.

**Cloud mode (signed in):**
4. Desktop app uses Convex SDK directly — React hooks (`useQuery`, `useMutation`) in the Tauri webview, no server routes in between.
5. Cloud workspace/session summary state uses Convex reactive queries with incremental refresh.
6. Organization switcher updates active `organizationId` applied to all subsequent Convex queries.
7. Activity feed reads from the `activity` table via a Convex reactive query on `workspace_id` — new events appear in real-time without polling.
8. Terminal (cloud): xterm.js in Tauri webview, connected via WebSocket to Cloudflare Sandbox. WebSocket client is [`partysocket`](https://docs.partykit.io/reference/partysocket-api/) — auto-reconnect with exponential backoff, event buffering during disconnection. No PartyKit server dependency; `partysocket` is used as a standalone WebSocket client.
9. Service list in right panel reads from `workspaceServices.list(workspaceId)` reactive query (cloud) or Tauri IPC (local). Share toggles and port overrides call `workspaceServices.update()` mutation (cloud) or Tauri command (local).
10. Workspace status is reactive via Convex subscriptions (cloud) or Tauri events (local).

### Non-Goals (V1)

1. Full IDE replacement.
2. Local workspaces visible to teammates (local workspaces are private; fork-to-cloud for collaboration).

## Acceptance Criteria

### Wedge Acceptance (Milestones 1-6 exit gate — must pass before expansion)

1. A net-new user can add a local project and reach a running workspace with terminal in under 5 minutes — no account required.
2. `lifecycle workspace create` reaches `ready` in under 60 seconds (p95, cloud) and under 30 seconds (p95, local).
3. First terminal session can be started and attached from desktop app workspace console in at most 2 user actions once workspace is `ready`.
4. First PR can be created from workspace branch through control-plane GitHub App permissions without requiring `gh` CLI in sandbox.
5. Health checks must pass before `ready`; failures surface service name + check type + error.
6. `workspace reset` restores deterministic baseline (post-setup state, data re-seeded).
7. `workspace service set --share on` returns URL usable by teammate with organization membership (cloud workspaces).
8. `workspace service set` supports validated per-workspace port overrides and share toggles.
9. `workspace destroy` terminates sandbox (cloud) or kills process group + prunes worktree (local), cleans up metadata.
10. TTL enforcement automatically expires and cleans up stale cloud workspaces.
11. Per-workspace mutation locking prevents concurrent conflicting operations.
12. Every failure surfaces a typed error with what happened, why, and suggested next action.
13. Local-first entities (`project`, `workspace`, `workspace_service`, `terminal`) have complete Tauri SQLite schema. Cloud-only entities (`organization`, `repository`, `activity`) have complete Convex schema. Sync between local and cloud is reliable when signed in.
