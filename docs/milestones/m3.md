# Milestone 3: "I can run an agent in a terminal"

> Prerequisites: M2
> Introduces: `terminal` entity, PTY architecture, harness integration

## Goal

User opens a terminal tab in a ready workspace, optionally selects a coding agent harness (Claude Code, Codex, etc.), and interacts with a real-time PTY session that supports detach/reattach.

## What You Build

1. xterm.js in Tauri webview, connected via Tauri IPC to `portable-pty`.
2. Terminal entity: start, detach, attach, finish lifecycle.
3. Harness support (Claude Code, Codex, etc.).
4. Terminal tabs with state indicators.
5. PTY output buffering for detach/reattach.

## Entity Contracts

### `terminal` (PTY session)

1. Purpose:
   - PTY process running inside a workspace; may be a plain shell or a coding agent session (`harness` being null = plain shell, set = agent session)
2. Required fields:
   - `id`
   - `workspace_id`
   - `harness` (nullable string, e.g. `"claude"`, `"codex"`, `"opencode"`, `"amp"`) — null for plain shell terminals
   - `harness_session_id` (nullable string; the agent CLI's own session/resumption token, captured at launch; co-null with `harness`)
   - `created_by` — user who started the session
   - `label` (human-friendly, e.g. "Terminal 1", "Claude · auth-fix")
   - `status` (`active|detached|sleeping|finished|failed`)
   - `failure_reason` (nullable typed enum; required when `status=failed`)
   - `started_at`, `last_active_at`
3. Invariants:
   - terminal attach only allowed when workspace is `ready`
   - sleeping workspaces force active/detached terminals to `sleeping`
   - workspace `destroy` hard-terminates any non-finished/non-failed terminal

## Implementation Contracts

### Terminal Model

A terminal session is a PTY process running inside a workspace execution environment. It may optionally run a coding agent harness (Claude Code, Codex, etc.) or be a plain shell. The `harness` field discriminates: null = plain shell, set = agent session. The desktop app terminal panel is a terminal client connected to the workspace PTY (via WebSocket for cloud, Tauri IPC for local); the process runs inside the execution environment (cloud sandbox or local host).

### Agent CLI Integration

1. **Claude Code**:
   - launch: `claude --session-id <uuid>` in workspace worktree directory inside workspace terminal
   - resume: `claude --resume <harness_session_id>` in same directory
   - capture ID: `--output-format json` emits `session_id` on first response
   - sessions are directory-scoped — worktree isolation is automatic
2. **Codex CLI**:
   - launch: `codex` in workspace worktree directory inside workspace terminal
   - resume: `codex resume <harness_session_id>`
   - capture ID: `--json` flag emits `thread.started` event with `thread_id`
   - sessions stored in `~/.codex/sessions/` as JSONL rollout files
3. **Other harnesses** (opencode, aider, amp, etc.):
   - generic PTY spawn in workspace terminal with no session capture
   - resume not supported; session is ephemeral

### Terminal Architecture

Part of `WorkspaceProvider.openTerminal()`. The terminal contract is provider-agnostic; the transport and process model vary by provider.

#### Contract (provider-agnostic)

1. PTY spawn in execution context with configurable shell and dimensions.
2. Attach/detach/reattach semantics: switching sessions preserves process state and buffers output for replay.
3. Multiple sessions per workspace supported as tabs.
4. On workspace destroy, all terminal sessions are terminated gracefully.

#### `LocalWorkspaceProvider` Terminal (M3 scope)

1. Tauri Rust backend spawns local child process via `portable-pty` (Rust PTY crate) in workspace worktree directory.
2. PTY output piped directly to xterm.js in the Tauri webview via Tauri IPC (no WebSocket intermediary).
3. CLI clients connect via Unix socket or Tauri CLI bridge.
4. Detach: process continues under Tauri supervision; output buffered for reattach.
5. Reattach: reconnects to Tauri-managed PTY session with buffered output replay.

#### `CloudWorkspaceProvider` Terminal (M4+ scope)

1. Lifecycle opens a Cloudflare sandbox terminal and spawns the process (shell or harness CLI) inside that PTY.
2. Terminal attach is authorized with short-lived, scoped attach tokens minted by Lifecycle control plane.
3. The desktop app terminal panel renders terminal output (xterm.js in Tauri webview) and forwards keyboard input over a WebSocket bridge.
4. Detach: user can switch tabs; process continues in sandbox.
5. Reattach: selecting a detached session reconnects to the same sandbox terminal stream with buffered output replay.

### Terminal State Machine

Full transition rules: [reference/state-machines.md](../reference/state-machines.md)

### Terminal Lifecycle

1. `start`: user creates a new terminal from workspace context, optionally selecting harness.
2. `detach`: user switches away; process keeps running in execution environment, output buffered.
3. `attach`: user selects terminal tab; PTY output streams to panel, input forwarded.
4. `sleep`: workspace sleeps; active/detached terminals transition to `sleeping`.
5. `wake + reattach`: workspace wakes and user reattaches; terminal returns to `detached` or `active` depending on whether a stream is attached.
6. `finish`: process exits naturally; terminal marked `finished`, transcript preserved.
7. `kill`: user explicitly terminates a terminal.

## Desktop App Surface

- **Terminal tabs**: tab bar with label (e.g., "Terminal 1", "Claude · auth-fix") and state indicators
- **"+" button**: create new terminal, choose harness or plain shell
- **xterm.js panel**: real-time PTY output rendering
- **Tab switching**: detach/attach without killing process
- **Finished state**: tab shows exit code when process completes
- **Theme system foundation**: support `appearance` (`light|dark|system`) and named preset themes (for future IDE-style themes)
- **UI primitives foundation**: establish shared `@lifecycle/ui` package for tokens and shadcn-ready reusable primitives

## Exit Gate

- Workspace is ready → click "+" to add terminal tab
- Choose "Terminal", "Claude Code", or "Codex"
- Terminal output streams in real-time
- Switch tabs → process keeps running → switch back → output replayed
- Process exits → tab shows finished state with exit code

## Test Scenarios

```
workspace ready → new terminal → shell prompt appears → type command → see output
workspace ready → new Claude Code session → agent starts → output streams
terminal active → switch to another tab → switch back → buffered output replays
terminal active → close tab → process terminates → tab shows "finished"
detach terminal → reopen app → terminal still listed → reattach → output continues
```
