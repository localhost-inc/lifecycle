# Milestone 3: "I can run an agent in a terminal"

> Prerequisites: M2
> Introduces: `terminal` entity, local PTY architecture, harness integration
> Tracker: high-level status/checklist lives in [`docs/plan.md`](../plan.md). This document is the detailed implementation contract.

## Goal

User opens a terminal tab in a created workspace, optionally selects a coding agent harness (Claude Code, Codex, etc.), and interacts with a real-time PTY session that supports detach/reattach within the running desktop app. Newly created workspaces should open with an initial harness tab so coding can begin immediately.

## What You Build

1. `@xterm/xterm` in the Tauri webview, with current addon packages (`@xterm/addon-fit`, `@xterm/addon-web-links`) and theme tokens from `@lifecycle/ui`.
2. Local `terminal` entity and Rust-side PTY supervisor owned by the Tauri backend.
3. Terminal transport split by responsibility:
   - Tauri `invoke` for control-plane operations (`create`, `attach`, `write`, `resize`, `detach`, `kill`)
   - Tauri `Channel` for ordered PTY output and attach replay
   - store events for terminal metadata/status changes only
4. Harness adapters for plain shell plus supported coding CLIs.
5. Bounded output buffering and replay for detach/reattach, with backpressure so the renderer does not fall behind.
6. Terminal tabs with live state powered by the desktop store/hook layer.

## Non-Goals (Explicit)

1. No external CLI attach bridge in M3. CLI-driven terminal control belongs in M6.
2. No cloud terminal transport in M3. Cloud workspace terminals belong in M7 when cloud workspaces exist.
3. No live local PTY survival across desktop app restart. The local PTY supervisor lives inside Tauri; if the app exits, local terminals exit too.
4. No requirement to support resumable session capture for every harness. Resume is per-harness capability, not a universal M3 gate.

## Entity Contracts

### `terminal` (PTY session)

1. Purpose:
   - PTY process running inside a workspace; may be a plain shell or a coding harness session (`launch_type=shell` means plain shell)
2. Required fields:
   - `id`
   - `workspace_id`
   - `launch_type` (`shell|harness|preset|command`) ã M3 uses `shell` and `harness`
   - `harness_provider` (nullable string, e.g. `"claude"`, `"codex"`) ã null unless `launch_type=harness`
   - `harness_session_id` (nullable opaque string) ã persisted only when a harness exposes a stable resumable session identifier
   - `created_by` (nullable for local pre-auth sessions)
   - `label` (human-friendly, e.g. `"Terminal 1"`, `"Claude ôñ auth-fix"`)
   - `status` (`active|detached|sleeping|finished|failed`)
   - `failure_reason` (nullable typed enum; required when `status=failed`)
   - `exit_code` (nullable integer)
   - `started_at`, `last_active_at`, `ended_at` (nullable; required when `status=finished|failed`)
3. Invariants:
   - create and attach are allowed whenever the parent workspace has an interactive worktree context
   - service sleep does not automatically suspend terminals; terminal access remains available while the worktree exists
   - workspace `destroy` hard-terminates any non-finished/non-failed terminal
   - `status=finished` requires `ended_at`; `exit_code` is recorded when the process exits with one
   - `launch_type=harness` requires `harness_provider`
   - `harness_session_id` is provider-owned metadata, not a cross-harness contract

## Implementation Contracts

### Terminal Model

A terminal session is a PTY process running inside a workspace execution environment. It may run a plain shell or a coding harness. The desktop terminal panel is a client attached to that PTY.

For local mode in M3:
- the PTY process runs on the host via the Tauri Rust backend
- the PTY working directory is the workspace `worktree_path`
- metadata and lifecycle state flow through the desktop store
- raw terminal output does **not** flow through the generic store event bus

### Reactive Data Model

1. Terminal metadata uses the same app-facing reactive substrate introduced in M2:
   - feature hooks such as `useWorkspaceTerminals()` and `useTerminal()`
   - store-backed query descriptors and reducers
   - coarse terminal lifecycle events (`terminal-created`, `terminal-status-changed`, `terminal-removed`)
2. Raw PTY output is a dedicated stream attachment, not a reducer-driven store event.
3. Terminal creation and lifecycle mutations stay imperative and transport-oriented.

### Harness Integration

1. Harness support is adapter-based:
   - `plain shell`
   - `Claude Code`
   - `Codex`
   - future CLIs without changing the terminal transport
2. M3 requires reliable PTY launch for supported harnesses. It does **not** require stable resumable session capture for every harness.
3. `harness_session_id` is optional and opaque:
   - set it when a harness exposes a stable documented resume token
   - leave it null when the harness does not expose one or the signal is not reliable enough
4. Claude Code has a documented `--resume` flow for existing sessions. Initial launch should be a normal interactive CLI invocation, not a synthetic app-assigned session id.
5. Codex and other harnesses should be treated as adapter-defined integrations. Do not hardcode milestone-critical behavior around undocumented or fast-moving CLI session internals.

### Provider Contract (M3 Scope)

The current M2 placeholder `openTerminal(workspaceId, cols, rows)` is insufficient for M3. The actual terminal surface should expand to explicit lifecycle operations:

1. `createTerminal(workspaceId, launchType, cols, rows, harnessProvider?, harnessSessionId?)` ã terminal metadata + initial attachment handle
2. `attachTerminal(terminalId, cols, rows)` ã ordered stream attachment + replay cursor
3. `writeTerminal(terminalId, data)` ã input bytes
4. `resizeTerminal(terminalId, cols, rows)` ã PTY resize
5. `detachTerminal(terminalId)` ã detach active client without killing the process
6. `killTerminal(terminalId)` ã terminate PTY session

### Local Terminal Architecture (M3 Scope)

1. Tauri Rust backend spawns the PTY with `portable-pty` in the workspace worktree directory.
2. PTY output is streamed over Tauri `Channel`, not over the coarse event bus.
3. Control operations use Tauri commands (`invoke`) so the command boundary stays typed and explicit.
4. The Rust terminal supervisor owns:
   - PTY handle
   - writer handle
   - current attach state
   - bounded replay buffer
   - exit status / failure status
5. Detach/reattach is supported only while the desktop app remains running.
6. App shutdown terminates local PTY sessions in M3 because the supervisor lives inside Tauri.
7. External CLI attach/detach is deferred to M6.

### Stream and Replay Contract

1. Stream PTY output as ordered chunks/bytes, not line-oriented messages.
2. Use bounded replay buffers in Rust so a newly attached client can catch up quickly.
3. Reattach replay should be batched and ordered before switching to live streaming.
4. Large output bursts must apply backpressure or chunk throttling so xterm stays responsive.
5. Terminal resize must propagate both directions cleanly:
   - frontend resize ã PTY resize
   - attach with cols/rows ã initial PTY dimensions

### Tab, Window, and Process Semantics

1. Switching away from a terminal tab detaches the UI client; it does not terminate the PTY process.
2. Closing a terminal tab in the UI must not silently kill a running process.
3. For `active` or `detached` terminals, the default close-tab behavior is `detach`, not `kill`.
4. Explicit terminate/kill is a separate user action and is the only normal tab-level action that ends a running PTY intentionally.
5. Reopening a previously closed terminal tab in the same desktop app session reattaches to the existing detached terminal if it still exists.
6. Natural process exit transitions terminal state to `finished`; terminated processes may resolve to `finished` or `failed` depending on exit semantics and recorded failure reason.
7. Desktop app quit/window close in M3 terminates local PTY sessions because the local terminal supervisor lives inside Tauri and no daemon-backed persistence exists yet.
8. Future daemon or tmux-backed persistence may change app-quit behavior, but that is explicitly out of scope for M3.

### xterm Client Contract

1. Use the current xterm package family:
   - `@xterm/xterm`
   - `@xterm/addon-fit`
   - `@xterm/addon-web-links`
   - optional later: `@xterm/addon-search`
2. Preserve Lifecycle theming by mapping semantic design tokens into xterm theme options.
3. Treat PTY output as untrusted terminal data:
   - render only through xterm
   - never inject terminal output into the DOM as HTML
4. Keep the terminal usable under sustained output:
   - batch writes
   - avoid per-line React state churn
   - detach stream lifetime from component rerender lifetime

### Cloud Terminal Architecture (Deferred)

Cloud terminal transport is out of scope for M3. M3 should keep the domain model and provider contract compatible with a future cloud implementation, but cloud attach transport, auth tokens, and websocket bridging belong in M7.

## Desktop App Surface

- **Terminal tabs**: tab bar with label and state indicators
- **Initial harness tab**: newly created workspaces open with a harness terminal automatically
- **New terminal action**: create plain shell or supported harness terminal from any workspace with interactive context
- **xterm panel**: real-time PTY rendering with resize support
- **Tab switching**: detach/attach without killing the process
- **Explicit terminate action**: user can kill a terminal intentionally
- **Finished state**: tab shows exit code when available

## Exit Gate

- Workspace is created ã first harness tab appears automatically
- Workspace has interactive context ã click `+` to add additional terminal tabs
- Choose plain shell or a supported harness
- Terminal output streams in real time
- Switch tabs ã process keeps running ã switch back in the same app session ã replay catches up, then live output continues
- Resize the panel ã PTY dimensions update correctly
- Explicitly terminate or let process exit ã tab shows finished state with exit code when available
