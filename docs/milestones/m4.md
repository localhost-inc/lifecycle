# Milestone 4: "I can run, reset, sleep, wake, and destroy a local workspace"

> Prerequisites: M3
> Introduces: run/reset semantics, local sleep/wake, local destroy, local preview, mutation locking, terminal/service separation
> Tracker: high-level status/checklist lives in [`docs/plan.md`](../plan.md). This document is the detailed implementation contract.

## Goal

The complete local workspace lifecycle loop works end-to-end: create -> run -> terminal -> reset -> sleep -> wake -> preview -> destroy -- with mutation locking and clear separation between agent terminals and service runtime state. No cloud, no auth, no network.

## What You Build

1. `run` (restart services) and `reset` (restore post-setup baseline).
2. Local sleep (SIGTERM process group) and wake (restart services, skip setup/clone).
3. Local destroy (kill process group, prune git worktree, clean SQLite).
4. Local preview (`localhost:<effective_port>`).
5. Service share toggles and port overrides in desktop UI (local-scoped).
6. Terminal access stays available while services are stopped, as long as the workspace worktree still exists.
7. Mutation locking hardened with typed errors.
8. Desktop app: full lifecycle controls, destruction confirmation, sleep/wake indicators.

## Implementation Contracts

### Run/Reset Semantics

- **`run`**: restart all workspace services using project-level `services` config plus workspace-level service overrides from `workspace_service` records.
- **`reset`**: restore baseline fixtures and restart all defined services. Uses `reset` config from `lifecycle.json` if present.

### Workspace Persistence Contract (Local)

**Mental model:** A Lifecycle workspace is a reproducible environment with optional hibernation, not a persistent VM. Every wake is a partial reconstruction. Reproducibility > persistence.

Part of `WorkspaceProvider.sleep()` / `.wake()`.

#### `LocalWorkspaceProvider` Persistence

What survives sleep:

| Resource                            | Survives sleep? | Mechanism                                        |
| ----------------------------------- | --------------- | ------------------------------------------------ |
| Filesystem / worktree               | Yes             | Already on disk -- no backup needed              |
| Git uncommitted changes             | Yes             | Already on disk                                  |
| Dependency cache                    | Yes             | Already on disk                                  |
| Docker images                       | Yes             | Docker Desktop manages image cache               |
| Docker volumes (e.g. Postgres data) | Yes             | Docker Desktop preserves volumes across restarts |
| Running processes                   | No              | SIGTERM process group on sleep; restart on wake  |
| Workspace metadata                  | Yes             | Stored in Tauri SQLite                           |

Sleep/wake contract:

1. Sleep: SIGTERM process group (services + containers). Record `sleeping` state. Worktree remains on disk.
2. Wake: restart services from manifest. Skip `setup` and `git clone`. Near-instantaneous.
3. Docker volumes survive sleep -- no re-seed needed unless explicitly requested via `reset`.

#### Wake vs Reset

- **Wake** = restore "where you left off." Filesystem preserved, all services restarted, no re-seed. Skips `setup` and `git clone`.
- **Reset** = restore "known-good baseline." Filesystem reset to post-setup state, data re-seeded, services restarted.

### Destroy Flow (Local)

- Kill process group, prune git worktree, clean SQLite metadata
- Workspace `destroy` hard-terminates any non-finished/non-failed terminal

### Live Preview Lifecycle (Local)

#### Contract (provider-agnostic)

1. **Scope**:
   - preview is defined per `workspace_service` record and represents routable access to an active service port inside a workspace execution environment
   - one workspace can expose multiple preview endpoints (for example `api`, `admin`, `docs`)
   - `exposure=local` enables localhost-only access

2. **Preview state machine** (`workspace_service.preview_state`):
   - Full transition rules: [reference/state-machines.md](../reference/state-machines.md)
   - The state machine is portable across providers -- trigger conditions apply regardless of mode

3. **Protocol support**:
   - HTTP/HTTPS and WebSocket (`ws/wss`) upgrade must be supported
   - Long polling and server-sent events must pass through without gateway buffering breakage
   - Raw TCP/UDP exposure is out of scope for preview URLs

4. **UX guarantees**:
   - preview URL remains stable for the life of the workspace (including hot reload and wake cycles)

#### `LocalWorkspaceProvider` Preview

1. **Routing model**:
   - `preview_url = http://localhost:<effective_port>`
   - `provisioning` state is near-instantaneous (port is already bound by the local process)
   - no proxy layer -- direct localhost access

2. **Access control**:
   - `exposure=local`: no auth required (localhost only)
   - `exposure=organization`: optional tunnel integration for sharing with teammates (deferred to expansion)

3. **Limitations**:
   - no wildcard DNS, no TLS by default
   - tunnel-based sharing (e.g., Cloudflare Tunnel, ngrok) is expansion-scope

### Mutation Concurrency

Full error catalog: [reference/errors.md](../reference/errors.md)

- Transitional states reject new workspace/service mutations with `workspace_mutation_locked` error
- Only `ready`, `sleeping`, and `failed` states accept mutation requests

### Terminal / Service Separation

- Workspace `sleeping` means services are stopped while the worktree remains available for interactive terminal work.
- Terminal create/attach stays available whenever the workspace has interactive context.
- Workspace `destroy` still hard-terminates any non-finished/non-failed terminal.
- Terminal state machine: [reference/state-machines.md](../reference/state-machines.md)

### SLOs

Full SLO targets: [reference/slos.md](../reference/slos.md)

Key M4 targets:
- p95 workspace wake from sleeping: <= 5s (local)
- p95 workspace create to `ready`: <= 30s (local)

## Desktop App Surface

- **Run button**: restart all services from manifest + overrides
- **Reset button**: restore post-setup baseline, re-seed, restart
- **Destroy button**: kill workspace with confirmation dialog
- **Reset/destroy confirmation**: explicit confirmation dialogs with workspace identifier
- **Sleep/wake indicators**: "sleeping" badge on workspace, auto-wake on click
- **Service share toggles**: per-service on/off toggle (local-scoped)
- **Port overrides**: editable port field per service
- **Preview URL display**: `localhost:<port>` with copy button

## Exit Gate

- Full local lifecycle loop works: create -> run -> terminal -> reset -> sleep -> wake -> preview -> destroy
- `run` restarts services without re-running setup
- `reset` restores post-setup baseline and re-seeds data
- Sleeping workspace shows "sleeping" badge -> click -> wakes automatically
- Destroy shows confirmation dialog -> confirms -> workspace gone, worktree pruned
- Terminal remains usable while services are sleeping
- Share toggle -> preview URL shows `localhost:<port>`

## Test Scenarios

```
workspace ready -> run -> services restart -> ready again
workspace ready -> reset -> services restart -> data re-seeded -> ready again
workspace ready -> sleep -> SIGTERM sent -> sleeping state -> wake -> services restart -> ready
workspace sleeping -> terminal remains usable -> wake -> services restart -> ready
destroy workspace -> confirmation dialog -> confirm -> workspace removed -> worktree pruned -> clean state
share service -> preview URL shows localhost:<port> -> opens in browser
mutation during transitional state -> workspace_mutation_locked error
```
