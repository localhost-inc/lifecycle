# Milestone 6: "Full lifecycle"

> Prerequisites: M5
> Introduces: run/reset semantics, sleep/wake, destroy, TTL enforcement, mutation locking

## Goal

The complete workspace lifecycle loop works end-to-end: create → run → terminal → reset → sleep → wake → share → destroy — on both providers, with TTL cleanup and hardened mutation locking.

## What You Build

1. `run` (restart services) and `reset` (restore post-setup baseline).
2. Sleep/wake for both local (SIGTERM/restart) and cloud (R2 backup/restore).
3. Destroy flow for both providers.
4. TTL enforcement + stale workspace cleanup (cloud).
5. Terminal sleep/wake transitions.
6. Mutation locking hardened with typed errors.
7. Desktop app: full lifecycle controls, destruction confirmation.

## Implementation Contracts

### Workspace Persistence Contract

**Mental model:** A Lifecycle workspace is a reproducible environment with optional hibernation, not a persistent VM. Every wake is a partial reconstruction. Reproducibility > persistence.

Part of `WorkspaceProvider.sleep()` / `.wake()`.

#### `CloudWorkspaceProvider` Persistence

What survives sleep:

| Resource                            | Survives sleep? | Mechanism                                   |
| ----------------------------------- | --------------- | ------------------------------------------- |
| Filesystem / worktree               | Yes             | R2 backup/restore                           |
| Git uncommitted changes             | Yes             | Auto-stash before sleep + filesystem backup |
| Dependency cache                    | Yes             | Separate R2 backup by lockfile hash         |
| Docker images                       | No              | Re-pull or restore from R2 cache on wake    |
| Docker volumes (e.g. Postgres data) | No (default)    | Re-seed on wake; opt-in backup for V2       |
| Running processes                   | No              | All services restart on wake                |
| Workspace metadata                  | Yes             | Stored in Convex                            |

Sleep/wake contract:

1. Sleep: back up worktree filesystem to R2. Terminate sandbox. Record `sleeping` state.
2. Wake: provision new sandbox, restore worktree from R2, re-run service startup (pull/start/health-check). Skip `setup` and `git clone`.
3. Docker data is NOT backed up in V1. Wake re-seeds if reset behavior is configured; otherwise services start with empty volumes.

#### `LocalWorkspaceProvider` Persistence

What survives sleep:

| Resource                            | Survives sleep? | Mechanism                                        |
| ----------------------------------- | --------------- | ------------------------------------------------ |
| Filesystem / worktree               | Yes             | Already on disk — no backup needed               |
| Git uncommitted changes             | Yes             | Already on disk                                  |
| Dependency cache                    | Yes             | Already on disk                                  |
| Docker images                       | Yes             | Docker Desktop manages image cache               |
| Docker volumes (e.g. Postgres data) | Yes             | Docker Desktop preserves volumes across restarts  |
| Running processes                   | No              | SIGTERM process group on sleep; restart on wake   |
| Workspace metadata                  | Yes             | Stored in Tauri SQLite (local) or Convex (cloud) |

Sleep/wake contract:

1. Sleep: SIGTERM process group (services + containers). Record `sleeping` state. Worktree remains on disk.
2. Wake: restart services from manifest. Skip `setup` and `git clone`. Near-instantaneous.
3. Docker volumes survive sleep — no re-seed needed unless explicitly requested via `reset`.

#### Wake vs Reset (both providers)

- **Wake** = restore "where you left off." Filesystem preserved, all services restarted, no re-seed. Skips `setup` and `git clone`.
- **Reset** = restore "known-good baseline." Filesystem reset to post-setup state, data re-seeded, services restarted.

#### Open Questions (Cloud)

1. Backup/restore performance at scale — 500MB worktree vs 2GB worktree R2 round-trip times.
2. Docker-in-Docker state after R2 restore — does `dockerd` recognize restored image layers, or must images be re-pulled?
3. Sandbox provisioning latency distribution — no published p50/p95 from Cloudflare for cold sandbox spin-up.
4. R2 egress costs for frequent wake operations — need cost model for teams with 20+ daily wake cycles.

### Run/Reset Semantics

- **`run`**: restart all workspace services using project-level `services` config plus workspace-level service overrides from `workspace_service` records.
- **`reset`**: restore baseline fixtures and restart all defined services. Uses `reset` config from `lifecycle.json` if present.

### Destroy Flow

- **Cloud**: terminate sandbox, revoke credentials, clean metadata
- **Local**: kill process group, prune git worktree, clean metadata
- Workspace `destroy` hard-terminates any non-finished/non-failed terminal

### TTL Enforcement

- TTL default: 24 hours
- Daily sweeper enforces expiration and marks final audit event
- Raw usage event retention: 13 months

### Mutation Concurrency

Full error catalog: [reference/errors.md](../reference/errors.md)

- Transitional states reject new mutations with `workspace_mutation_locked` error
- Only `ready`, `sleeping`, and `failed` states accept mutation requests
- Convex OCC handles serialization natively
- Global quotas by organization (max active workspaces)

### Terminal Sleep/Wake

- Workspace sleep forces active/detached terminals to `sleeping`
- Wake + reattach: terminal returns to `detached` or `active` depending on stream attachment
- Terminal state machine: [reference/state-machines.md](../reference/state-machines.md)

### SLOs

Full SLO targets: [reference/slos.md](../reference/slos.md)

Key M6 targets:
- p95 workspace wake from sleeping: <= 15s (cloud), <= 5s (local)
- p95 workspace create to `ready`: <= 60s (cloud), <= 30s (local)

## Desktop App Surface

- **Full lifecycle controls**: run, reset, destroy buttons in right operations panel
- **Destroy confirmation**: explicit confirmation dialog with workspace identifier
- **Reset confirmation**: explicit confirmation before execution
- **Sleep/wake indicators**: "sleeping" badge on workspace, auto-wake on click
- **Terminal sleep states**: tabs show "suspended" state during workspace sleep

## Exit Gate

- Full lifecycle loop works: create → run → terminal → reset → sleep → wake → share → destroy
- Sleeping workspace shows "sleeping" badge → click → wakes automatically
- Destroy shows confirmation dialog → confirms → workspace gone
- Stale cloud workspaces auto-cleaned

## Test Scenarios

```
workspace ready → reset → services restart → data re-seeded → ready again
workspace ready → idle timeout → sleeps automatically → click → wakes → ready
workspace sleeping → terminal tabs show "suspended" → wake → terminals resume or show "expired"
destroy workspace → confirmation dialog → confirm → workspace removed → clean state
cloud workspace idle 7 days → auto-destroyed by cleanup
```
