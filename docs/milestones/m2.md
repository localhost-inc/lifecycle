# Milestone 2: "I can start a workspace and it reaches ready"

> Prerequisites: M1
> Introduces: `workspace` entity, `workspace_service` entity, `LocalWorkspaceProvider` impl, workspace state machine

## Goal

User clicks "Start" on a project and watches a local workspace progress through `creating → starting → ready` with per-service health indicators.

## What You Build

1. Local workspace creation (git worktree checkout).
2. Tauri Rust backend: process supervisor for service lifecycle.
3. Docker Desktop integration for `image` runtime services.
4. Health check gate (tcp + http) → ready transition.
5. Setup step execution (sequential, exactly-once).
6. Desktop app: workspace status display with service health indicators.

## Entity Contracts

### `workspace` (execution environment)

1. Purpose:
   - isolated execution environment at one ref/SHA
   - always scoped to a `project` — regardless of mode
2. Required fields:
   - `id`
   - `project_id`
   - `source_ref`
   - `git_sha`
   - `worktree_path` — relative path within execution context; absolute path is derived at runtime by the workspace provider
   - `mode` (`cloud|local`) — which `WorkspaceProvider` runs this workspace
   - `mode_state` (JSONB, nullable) — mode-specific runtime state (see table below)
   - `status` (`creating|starting|ready|resetting|sleeping|destroying|failed`)
   - `failure_reason` (nullable typed enum; required when `status=failed`)
   - `failed_at` (nullable timestamp; required when `status=failed`)
   - `created_by` (nullable — not set for local workspaces pre-auth)
   - `created_at`, `updated_at`, `last_active_at`, `expires_at`
   - `source_workspace_id` (nullable UUID) — if this workspace was created via fork, references the originating workspace

   **`mode_state` shape by mode:**

   | Mode | State shape |
   |-------------|---------------|
   | `cloud` | `{ sandbox_id, sandbox_status, r2_backup_key, capacity_pool }` |
   | `local` | `{ process_group_id }` |

   - workspace display name is derived from `source_ref` (e.g., branch name `feature/auth-fix` → "auth-fix"); no separate `name` field in V1

3. Workspace invariants:
   - transitional status acts as implicit mutation lock — no concurrent mutations allowed
   - workspace state machine is authoritative for allowed actions

### `workspace_service` (per-service runtime state)

1. Purpose:
   - runtime state for a single service within a workspace; extracted from embedded map to eliminate OCC contention between concurrent service updates
2. Required fields:
   - `id`
   - `workspace_id`
   - `service_name` — key from project `lifecycle.json` services config
   - Writable fields (accepted by PATCH):
     - `exposure` (`internal|organization|local`)
     - `port_override` (nullable)
   - Computed/runtime fields:
     - `status` (`stopped|starting|ready|failed`)
     - `status_reason` (nullable typed enum; required when `status=failed`)
     - `default_port` (nullable) — from project `lifecycle.json` config
     - `effective_port` — computed: `coalesce(port_override, default_port)`
     - `preview_state` (`disabled|provisioning|ready|sleeping|failed|expired`)
     - `preview_failure_reason` (nullable typed enum; required when `preview_state=failed`)
     - `preview_url` (nullable)
3. Invariants:
   - unique (`workspace_id`, `service_name`)
   - `exposure=organization` requires an effective port
   - `exposure=local` is valid only when parent `workspace.mode=local`; rejected for `cloud` workspaces
   - `preview_url` must be null unless `preview_state=ready`
   - `preview_state=ready` requires `exposure=organization` or `exposure=local`

## Implementation Contracts

### Workspace State Machine

Full transition rules: [reference/state-machines.md](../reference/state-machines.md)

M2-relevant state semantics (local mode):

| State | `LocalWorkspaceProvider` behavior |
|-------|----------------------------------|
| `creating` | Creating git worktree + running setup |
| `starting` | Starting local processes + containers |
| `ready` | Health checks pass on localhost |
| `failed` | Typed `failure_reason` + `failed_at` recorded |

### `WorkspaceProvider` Interface

Full interface: [reference/workspace-provider.md](../reference/workspace-provider.md)

M2 implements `LocalWorkspaceProvider`:
- `createWorkspace`: git worktree checkout + setup step execution
- `startServices`: spawn local processes + Docker containers
- `healthCheck`: tcp/http probes against localhost
- `stopServices`: SIGTERM process group

### Health Check Contract

- Per-service `health_check` object with `type` (`tcp` or `http`)
- Workspace transitions to `ready` only after all defined service health checks pass
- Full spec: [reference/lifecycle-json.md](../reference/lifecycle-json.md)

### Workspace Topology

- One workspace == one execution environment logically mapped to one Git worktree path
- Lifecycle manages worktree create/prune to avoid branch checkout collisions

### Testing Experience Design

1. Project runtime config (`lifecycle.json`) drives execution:
   - `setup` steps for new workspace creation
   - `services` for long-lived runtime start/restart
   - fixture/seed behavior
2. Fast reset path: deterministic seed and clean state restore
3. Test-first UX: one keypath for `create -> test -> share results`
4. Output quality: real-time logs, summarized failures with links to full logs

### Reliability Baseline

1. Workspace mutations use status field as implicit lock — failed transitions surface typed errors. Local workspaces use SQLite transactions.
2. Workspace health checks (service readiness + probe endpoints).
3. Daily cleanup of stale workspaces.

## Desktop App Surface

- **Workspace status display**: creating → starting → ready progression
- **Per-service health indicators**: spinner → green checkmark
- **Setup step progress**: sequential progress with stdout
- **Ready state**: all services green, workspace badge "ready"

## Exit Gate

- You click "Start" on your project
- Watch progress: creating → starting → ready
- Each service shows health status (spinner → green checkmark)
- Setup steps show sequential progress with stdout
- When ready, all services green, workspace badge says "ready"

## Test Scenarios

```
select project → click Start → watch creating→starting→ready transitions
services with health_check show individual status progression
setup step failure → workspace shows failed with step name + error output
service health check failure → workspace shows failed with service name + check type
stop workspace → services shut down → workspace transitions to sleeping
```
